    Matching

    The matching module we import from npm requires us to specify how it should interact with the mathematical expressions in the client project. We make a call to the matching module's setAPI() function and provide it with all the tools it needs to interface with LogicConcept instances. We document in this namespace all those tools.

    Methods

    static

    application(children) → {Application}

    How can the matching module construct a function application with a given set of expressions to use as children? (By convention, the first is the operator/function being applied, and the rest are its arguments, in order.) We simply call the Application constructor.

    Note that all children provided in the first argument will be deep copied, because the matching module expects and relies upon that behavior. It is not possible, using this function, to move/share subexpressions between expressions.

    Parameters

    • children Array.<Expression>

      a JavaScript array of the child expressions, in the order described above

    Returns

    • Application

      an instance of the Application class, which this function creates

    getChildren(expression) → {Array.<Expression>}

    How can the matching module query the list of children expression of a given expression? The MathConcept class provides the children() function for this purpose.

    A few notes about it:

    • An Application's children are returned in the order explained here.
    • A Binding's children are its head, then its list of bound variables (Symbol instances), in order, then its body.
    • An atomic expression will have an empty children list.

    Parameters

    • expression Expression

      the expression whose children are being requested

    Returns

    • Array.<Expression>

      the JavaScript array of all children of this expression

    symbol(text) → {Symbol}

    How can the matching module construct a variable with a given name? We use the Symbol class to store both what the matching module calls "variables" and what it calls "symbols," but we store variables with whatever name is requested, whereas we store symbols with the prefix "symbol: " in front of their name, to distinguish them from variables.

    While there is a small chance that this could cause ambiguity if you have a variable whose name literally begins with the text "symbol: ", that event is unlikely in the extreme, and a bad idea anyway.

    Parameters

    • text string

      the name to use for the symbol being created; this text will be prefixed with "symbol: "

    Returns

    • Symbol

      an instance of the Symbol class, the symbol that was created

    Parameters

    • variable Symbol

      the variable to test for freeness; note that this is an actual Symbol instance whose ancestor context is relevant to this function, not just the name or another copy of the same variable

    • expression Expression <optional>

      optional ancestor context in which to test for freeness; if this is provided, only bindings within this ancestor are relevant, but if it is omitted, all ancestor bindings are relevant

    Returns

      whether the given variable is free in the given ancestor expression (or its topmost ancestor if no ancestor expression is provided)
